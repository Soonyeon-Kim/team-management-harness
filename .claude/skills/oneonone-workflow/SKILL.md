# 1:1 미팅 워크플로우

1:1은 팀원 관리의 신경계다. 단발 대화가 아니라 **이전 액션 → 이번 논의 → 다음 액션**으로 이어지는 루프다. 이 연속성을 깨면 팀원은 "말해봤자 안 바뀐다"고 느낀다. 따라서 모든 작업은 이전 기록을 먼저 읽는 데서 시작한다.

> 데이터 위치·스키마는 `team-data-store` 스킬을 따른다. 1:1 기록은 `team-data/oneonones/{slug}/YYYY-MM-DD.md`.

## 팀 루트 해석 (작업 전 필수)

이 워크플로우가 참조하는 모든 `team-data/...` 경로는 **팀 루트(`team_root`) 기준**이다. 데이터 접근 전 `team-data-store`의 "멀티-팀 모드 (팀 루트 해석)"를 적용한다:
- `team-data/teams/`가 없으면 → `team_root = team-data/` (레거시 단일 팀, 기존 동작 유지).
- 있으면 → `team_root = team-data/teams/{team_id}/`. `team_id`(또는 팀명)는 오케스트레이터가 전달하고, 팀이 1개면 자동 선택.
- 본문의 `team-data/<x>`는 모두 `{team_root}/<x>`로 읽고 쓴다(1:1 기록 `oneonones/{slug}/...` 포함).
- **전사 공통(항상 루트, 예외):** 공휴일 `team-data/calendar/holidays.json`.

## 작업 유형

### 1. 1:1 준비 (아젠다 생성)
1. 대상 팀원의 `members/{slug}.md`, 가장 최근 `oneonones/{slug}/*.md`, `growth/{slug}.md`를 읽는다.
2. 지난 1:1의 **미완료 액션 아이템**을 추출한다 — 이게 이번 아젠다 1순위다.
3. 팀원의 최근 컨디션(`mood`), 성장 목표, 진행 중 업무(task-tracker 데이터가 주어지면 참조)를 엮어 논의 주제 3~5개를 제안한다.
4. 출력: 아젠다 초안. 팀장이 추가/삭제할 수 있도록 제시한다. **이 단계에서는 파일을 쓰지 않는다**(미팅 전이므로).

### 2. 1:1 기록
1. 팀장이 제공한 미팅 내용을 받는다.
2. `oneonones/{slug}/{오늘}.md`를 스키마대로 생성한다: 논의 주제, 팀원 발언 요약, 액션 아이템(담당·마감 명시), 팔로업(지난 액션 점검 결과).
3. `mood`를 팀장 판단에 따라 기록한다.
4. 액션 아이템 중 업무성 항목이 있으면 "task-tracker에 등록할까요?"라고 제안한다(직접 tasks.json을 건드리지 않는다 — 도메인 경계 존중).

### 3. 팔로업 점검
- 특정 팀원 또는 전체의 미완료 1:1 액션 아이템을 스캔해 보고한다.
- 마지막 1:1 이후 경과일을 계산해 **체크인 주기를 넘긴 팀원**을 식별한다(team.md cadence 참조, 기본 격주 2주).

## 핵심 규칙

- **이전 기록 우선**: 준비든 기록이든 항상 직전 1:1을 먼저 읽는다. 팔로업 없는 1:1은 만들지 마라.
- **액션 아이템은 검증 가능하게**: "성장에 신경쓰기"(X) → "다음 주까지 React 강의 1챕터 완료, 1:1에서 공유"(O). 담당과 마감이 없으면 추적이 불가능하다.
- **경계 존중**: 업무 등록은 task-tracker, 성장 목표 갱신은 growth-coach, 휴가는 leave-manager의 영역이다. 제안만 하고 해당 도메인 파일을 직접 쓰지 마라. 이유: 데이터 무결성과 책임 분리.
- **민감 정보**: 팀원 발언은 사실 위주로 요약하고, 팀장의 주관적 평가는 별도로 표시한다.

## 이전 산출물이 있을 때

- 같은 날짜 기록이 이미 있으면 덮어쓰지 말고 읽어서 병합/보완한다.
- 팀장이 "방금 1:1 다시 정리"라고 하면 가장 최근 파일을 수정 대상으로 삼는다.
