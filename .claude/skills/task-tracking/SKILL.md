# 업무/태스크 추적 워크플로우

업무 추적의 목적은 일감 목록 보관이 아니라 **위험 조기 발견**이다 — 마감 임박, 지연, blocked, 특정 팀원 과부하. 단순 CRUD로 끝내지 말고 항상 위험 신호를 함께 보고한다.

> 데이터 위치·스키마는 `team-data-store` 스킬을 따른다. 업무는 `team-data/tasks/tasks.json`.

## 작업 유형

### 1. 업무 등록
1. `tasks.json`을 읽고 새 ID(`T-{최대값+1}`)를 부여한다.
2. `assignee`가 `members/`에 존재하는 slug인지 확인한다. 없으면 등록하지 말고 보고한다(임의로 팀원 생성 금지).
3. 필수 필드(title, assignee, status, priority, due, created, updated)를 채워 배열에 추가한다. due가 불명확하면 팀장에게 묻는다.

### 2. 상태/필드 갱신
1. 대상 업무를 ID 또는 title로 찾는다.
2. read-modify-write로 해당 필드만 바꾸고 `updated`를 오늘로 갱신한다. 다른 업무는 건드리지 않는다.
3. `done` 처리 시 완료일을 notes에 남긴다.

### 3. 현황/위험 리포트
다음을 계산해 보고한다:
- **지연(overdue)**: `due < 오늘` 이고 `status != done`.
- **마감 임박**: `due`가 오늘~3일 이내.
- **blocked**: `status == blocked` (얼마나 오래 막혔는지 `updated` 기준).
- **팀원별 부하**: assignee별 미완료(todo+in-progress+blocked) 개수. 한 명에 과집중되면 플래그.

출력은 위험도 높은 순으로 정렬한다(overdue → 임박 → blocked → 일반).

## 핵심 규칙

- **위험 우선 보고**: 단순히 목록을 나열하지 말고 "지금 챙겨야 할 것"을 맨 위에 둔다.
- **assignee 무결성**: 존재하지 않는 팀원에게 할당하지 마라. 휴가 중(`status: on-leave`)인 팀원에게 마감 임박 업무가 있으면 보고에 명시한다(leave-manager 데이터와의 충돌 신호).
- **경계 존중**: 휴가 등록은 leave-manager, 1:1 액션은 oneonone-coach 영역이다. 업무성 액션 아이템을 1:1에서 넘겨받아 등록하는 것은 가능하나, 출처(어느 1:1에서 나왔는지)를 notes에 남긴다.
- **추정 금지**: 진행률을 임의로 만들지 마라. 팀장/팀원이 제공한 status만 기록한다.

## 이전 산출물이 있을 때

- "업무 현황 다시" 요청 시 tasks.json을 다시 읽어 최신 상태로 리포트한다(캐시된 이전 결과를 재사용하지 마라 — 그사이 변경됐을 수 있다).
