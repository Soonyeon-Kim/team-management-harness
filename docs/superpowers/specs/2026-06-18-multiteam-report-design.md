# 멀티-팀원 / 멀티-팀 상급자 보고 리포트 설계

**작성일:** 2026-06-18
**대상:** `exec-report-web` 스킬, `team-data/` 데이터 모델, 도메인 스킬/에이전트 6종
**선행 문서:** `2026-06-17-exec-report-redesign-design.md`(단일 팀 모던 SaaS 디자인 개편), `docs/superpowers/plans/2026-06-17-exec-report-redesign.md`(미실행 계획)

---

## 1. 배경 / 문제

상급자 보고 HTML 리포트는 현재 **단일 팀 전제**다.

- 데이터: `team-data/` 루트가 암묵적으로 한 팀(`team.md` = DS팀/김수년). 멀티-팀 모델링 없음.
- 리포트: 팀원은 `member` REPEAT 블록으로 N명 지원되나, 다인원에서 (a) 표 가로 스크롤, (b) 어느 팀원이 급한지 안 보임, (c) 위험 Top 3~5 무음 절단, (d) KPI 평균이 이상치 은폐 — 가독성 갭이 있다.

요구: **(요청1)** 멀티-팀원 가독성을 개편 계획에 반영. **(요청2)** 여러 동격 팀이 데이터에 상존할 때의 표현 설계.

## 2. 확정된 결정 (브레인스토밍)

| 결정 | 값 |
|------|-----|
| 조직 구조 | **동격 팀 다수** — 한 도구로 독립 팀들을 각각 관리(팀장 다를 수 있음). 기본 팀 단위 보고 + 필요 시 조직 통합본. |
| 범위 | **풀 구현** — 데이터 모델 + 팀 스코프 리포트 + 조직 통합본 모두. |
| 통합본 형태 | **요약 + 드릴다운 링크**(조직 요약 한 장 → 각 팀 리포트로 이동). |
| 드릴다운 구현 | **팀별 별도 파일** — 팀마다 자체완결 .html, 조직 요약이 상대경로로 링크. 팀별 개별 공유 우선. |

## 3. 전체 구조 — 2트랙

요청의 무게가 달라 두 트랙으로 분리한다.

- **트랙 A (멀티-팀원, 저위험):** 기존 개편 계획 `2026-06-17-exec-report-redesign.md`의 Task 1에 가독성 항목 추가. exec-report-web에 국한.
- **트랙 B (멀티-팀, 대규모):** 본 문서의 신규 설계. 별도 구현 계획으로 작성. 데이터 모델 변경이 leave·cert·assignment·growth·oneonone 전 스킬/에이전트로 파급되므로 트랙 A와 분리.

**실행 순서:** 트랙 A 먼저(이미 거의 완성된 계획) → 트랙 B를 독립 스펙/계획으로. 단 트랙 A 템플릿 작성 시 트랙 B(팀 스코프·조직 요약)와 토큰/구조가 호환되도록 설계한다.

## 4. 트랙 A — 멀티-팀원 가독성

기존 단일 팀 리포트 구조(헤더·KPI 5·위험·팀원표·영역요약)는 유지하고, 다인원 견고성만 보강한다.

### 4.1 처방

| 갭 | 처방 |
|----|------|
| 표 가로 스크롤 (min-width 680px) | 컬럼 패딩·폰트 축소로 밀도 ↑. `<thead>` **sticky 헤더**(`position:sticky; top:0`)로 긴 표 스크롤 시 컬럼명 고정. |
| 다인원 시 우선순위 불가시 | 팀원 행을 **주의 신호 심각도순 정렬**(고심각 → 상단). 고심각 팀원 행에 좌측 강조(`tr.attn` 좌측 컬러 보더). |
| 위험 Top 3~5 무음 절단 | 잘릴 때 **"외 N건"** 명시(무음 절단 금지). 인원 많으면 노출 개수 상향 허용(스킬 지시문에 규칙화). |
| KPI 평균이 이상치 은폐 | KPI note에 **편차/이상치** 병기(예: "평균 78% · 최저 42% 박민호"). |

### 4.2 토큰 계약 변경 (트랙 A)

기존 토큰은 보존하고 **추가만** 한다.

- `member` 블록: `{{M_ATTN_CLASS}}` 추가 — 행 강조 클래스(빈값=일반, `attn`=주의 강조). `<tr class="{{M_ATTN_CLASS}}">`.
- `risk` 섹션: `{{RISK_MORE_NOTE}}` 추가(선택) — 절단 시 "외 N건" 문구. 없으면 빈값.

> 위 변경은 트랙 A 템플릿(`template.html`)과 `SKILL.md` 치환 규칙에 반영하며, **기존 개편 계획서의 Task 1·Task 2에 흡수**된다.

## 5. 트랙 B — 멀티-팀 데이터 모델

### 5.1 디렉터리 구조 (팀 스코프 하위 디렉터리)

태그 필드 방식 대신 **팀별 하위 디렉터리**로 격리한다(이행·격리·읽기 단순).

```
team-data/
  teams/
    ds/                  ← team-id (슬러그)
      team.md            (team: DS팀, lead: 김수년, cadence: weekly)
      members/  oneonones/  tasks/  growth/
      leave/    certifications/     assignments/
    ai/
      team.md            (team: AI팀, lead: 이지은)
      ... (동일 하위 구조)
  calendar/holidays.json ← 전사 공통(공유, 팀 밖)
  checklist.md           ← 팀장 체크리스트(공통 유지)
  _reports/
    2026-06-18-org-summary.html       ← 조직 요약(트랙 B §7)
    ds/2026-06-18-exec-report.html    ← 팀별 리포트(트랙 B §6)
    ai/2026-06-18-exec-report.html
```

- **team-id(슬러그):** 영소문자/하이픈(`ds`, `ai`, `consulting`). `team.md` frontmatter의 `team`(표시명)·`lead`와 연결.
- **공통 데이터:** `calendar/holidays.json`(전사 공휴일), `checklist.md`(팀장 체크리스트)는 팀 밖 루트에 유지.

### 5.2 하위 호환 & 마이그레이션

- **모드 판별:** `team-data/teams/`가 존재하면 **멀티-팀 모드**, 없으면 **레거시 단일-팀 모드**(현재처럼 루트 = 그 한 팀). 기존 동작 불변.
- **마이그레이션(1회):** 현재 루트의 팀 데이터(`members/ leave/ certifications/ assignments/ growth/ oneonones/ tasks/ team.md`)를 `teams/ds/`로 이동. `calendar/`·`checklist.md`·`_reports/`는 루트 유지.
- 마이그레이션은 트랙 B 구현 계획의 첫 Task로 두고, 이동 후 레거시 경로 잔존 여부를 grep으로 검증.

### 5.3 팀 해석 규칙 (team-data-store 스키마 보강)

- 도메인 스킬/에이전트는 데이터 접근 전 **"팀 루트 해석" 프리앰블**을 수행:
  1. `teams/`가 없으면 루트를 팀 루트로 사용(레거시).
  2. `teams/`가 있으면 호출 시 받은 `team` 인자(슬러그 또는 표시명)로 `teams/<id>/`를 선택.
  3. 팀이 1개면 자동 선택, 다수인데 인자 없으면 오케스트레이터(team-management)가 대상 팀을 확정하거나 "전체"를 명시.
- `team-data-store/SKILL.md`에 위 규칙과 새 레이아웃을 단일 표준으로 명문화. 6개 도메인 스킬(`leave/cert/assignment/growth/oneonone/task`)과 에이전트는 이 프리앰블을 참조하도록 보강.

> ⚠️ 비용 집중 지점: 데이터 모델 변경 자체보다 **6개 도메인 스킬/에이전트의 "팀 인식" 보강**이 트랙 B 작업량의 대부분이다. 구현 계획에서 별도 Task 군으로 분리한다.

## 6. 트랙 B — 팀별 리포트 스코프

- `exec-report-web`이 **팀 스코프 인자**(`team`)를 받는다.
- 출력 경로: `team-data/_reports/<team-id>/{YYYY-MM-DD}-exec-report.html`(`<team-id>/`가 없으면 생성).
- 내용·산식·토큰은 기존 단일 팀 리포트와 동일하되, **그 팀으로 스코프**한 데이터로 채운다. 각 파일은 자체완결 → 팀별 따로 공유 가능.
- 레거시(단일 팀) 호출은 기존 경로 `_reports/{YYYY-MM-DD}-exec-report.html` 유지(하위 호환).

## 7. 트랙 B — 조직 요약 + 드릴다운

### 7.1 산출물

- 신규 템플릿 `org-template.html`을 **exec-report-web 스킬 내 두 번째 템플릿**으로 추가(별도 스킬 아님).
- `SKILL.md`에 "조직 요약 생성" 절차를 추가: 멀티-팀 모드에서 각 팀 리포트 생성 후 조직 요약을 산출.
- 출력 경로: `team-data/_reports/{YYYY-MM-DD}-org-summary.html`.

### 7.2 조직 요약 구성

1. **헤더:** 조직명/기준일/내부 보고용 배지.
2. **조직 KPI:** 전팀 합산/평균(총 인원, 평균 가동률, 평균 연차 누적률, 자격증 합계, 1:1 due 합).
3. **팀 비교 미니표:** 행=팀. 열=팀명·팀장·인원·가동률·연차 누적률·자격증(보유/목표)·1:1 due. 가동률/연차는 막대색(ok/warn/bad), 이상치 플래그(⚠).
4. **조직 Top 위험:** 전팀 위험을 심각도순 취합, 각 항목에 **팀명 태그**. 절단 시 "외 N건".
5. **팀 바로가기:** 각 팀 리포트로 **상대경로 링크**(`./<team-id>/{YYYY-MM-DD}-exec-report.html`).

### 7.3 자체완결 유지

- 조직 요약 파일도 외부 의존성 0(인라인 CSS, CSS 막대). 링크는 `<a href>` 상대경로뿐.
- 팀별 파일은 각각 자체완결 → 팀 하나만 떼어 공유 가능. 조직 전체 공유는 `_reports/` 폴더(또는 ZIP) 단위(드릴다운 철학상 의도된 트레이드오프).
- 조직 KPI 산식은 새로 발명하지 않고 §exec-report-web KPI 산식의 팀별 값을 집계(평균/합산)만 한다.

## 8. 견고성 / 엣지 케이스

- **팀 0개/1개:** `teams/` 없음 = 레거시 단일 팀(현 동작). 팀 1개만 있는 멀티-팀 모드에서도 조직 요약이 깨지지 않음(미니표 1행, 링크 1개).
- **팀원 0명 팀:** 해당 팀 리포트는 "팀원 없음" empty, 조직 미니표는 인원 0·가동률 "—".
- **데이터 미입력:** 기존 규칙대로 "데이터 미입력"을 위험으로 과장하지 않음.
- **팀명/슬러그 불일치, 존재하지 않는 팀 참조:** 임의 보정 없이 오케스트레이터에 보고.
- **대규모 팀(다인원):** 트랙 A의 sticky 헤더·정렬·"외 N건"이 팀별 리포트에 그대로 적용.

## 9. 검증 (테스트 러너 없음 → grep + 육안)

- **트랙 A:** 토큰/마커 보존 grep, `M_ATTN_CLASS`·`RISK_MORE_NOTE` 존재, 외부 의존성 0 grep, 다인원(예 8명) 프리뷰 육안(sticky·정렬·절단 표기).
- **트랙 B:** 마이그레이션 후 레거시 경로 잔존 grep, 멀티-팀 모드에서 팀별 리포트 N개 + 조직 요약 1개 생성, 상대 링크 클릭 시 해당 팀 파일 열림(육안), 자체완결(외부 URL 0) grep, 레거시 단일-팀 호출이 기존 경로로 정상 산출(하위 호환).

## 11. 비범위 (YAGNI)

- 팀 간 권한/접근제어, 팀원의 팀 이동 이력, 조직 계층 3단계+ 롤업(본부→실→팀)은 이번 범위 밖. 현재는 동격 팀 평면 구조만.
- 실행 코드(JS)·서버·빌드 없음(코드 산출물 없는 Claude 네이티브 툴 원칙 유지).
